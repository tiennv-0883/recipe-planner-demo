# Implementation Plan: Multi-Language Support

**Branch**: `006-multi-language` | **Date**: 2026-03-31 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/006-multi-language/spec.md`

## Summary

Add full Vietnamese/English UI translation to the Recipe Planner app using **next-intl** with cookie-based locale detection (`localePrefix: 'never'`). Vietnamese is the default locale; English is the secondary. All hardcoded UI strings across 25 files are replaced with `useTranslations()` calls backed by `messages/vi.json` and `messages/en.json`. A `LanguageSwitcher` component in the sidebar sets the `NEXT_LOCALE` cookie via a Server Action and refreshes the RSC tree — no URL changes, no page reload, persistent across sessions.

## Technical Context

**Language/Version**: TypeScript 5 / Node.js 20 / Next.js 15.2.2 (App Router)  
**Primary Dependencies**: `next-intl` (to install); existing Supabase, Tailwind CSS v4, React 19  
**Storage**: `NEXT_LOCALE` HTTP cookie (1-year maxAge, path `/`, sameSite `lax`) — no DB changes  
**Testing**: Jest 29 + React Testing Library — existing 116 tests may need string literal updates; no new i18n-specific business-logic tests required  
**Target Platform**: Next.js 15 App Router on Vercel / local dev  
**Project Type**: Web application (monorepo single project)  
**Performance Goals**: Language switch re-renders current page in <300 ms (router.refresh() RSC re-fetch, typical LAN latency)  
**Constraints**: Supabase auth middleware (`src/middleware.ts`) MUST NOT be modified; `localePrefix: 'never'` is mandatory (FR-009); both locales must be fully covered (SC-005)  
**Scale/Scope**: 2 locales, 8 namespaces, ~112 translation keys, 25 files with hardcoded strings

## Phase 1: Design Artifacts

> All artifacts generated. See linked files for full content.

### Data Model — [data-model.md](./data-model.md)

**Entities** (no DB schema changes):

| Entity | Storage | Notes |
|--------|---------|-------|
| `Locale` | In-memory constant | `'vi' \| 'en'`; default `'vi'` |
| `MessageBundle` | `messages/{locale}.json` | 8 namespace keys, ~112 translation pairs |
| `LanguagePreference` | `NEXT_LOCALE` cookie | 1-year maxAge, sameSite `lax`, path `/` |

**Validation**: Cookie values not in `['vi', 'en']` are treated as absent → fallback to `'vi'`.

### Contracts — [contracts/translations.md](./contracts/translations.md)

**Key naming rules**:
- camelCase at all levels; max 3 levels of nesting
- Namespace = domain module (one namespace per module, not per component)
- ICU message syntax for plurals: `{count, plural, one {# item} other {# items}}`
- No locale codes in key names; no pluralization suffixes

**8 namespaces**: `auth` · `nav` · `dashboard` · `recipes` · `mealPlanner` · `groceryList` · `categories` · `notFound` · `common`

### Quickstart — [quickstart.md](./quickstart.md)

Key steps:
1. `npm install next-intl`
2. Create `src/i18n/config.ts`, `request.ts`, `routing.ts`
3. Wrap `next.config.ts` with `createNextIntlPlugin`
4. Create `messages/vi.json` + `messages/en.json`
5. Update `src/app/layout.tsx` to serve `NextIntlClientProvider`
6. Create `src/app/actions/locale.ts` (Server Action)
7. Create `src/components/LanguageSwitcher.tsx`
8. Replace all hardcoded strings in 25 files with `useTranslations()`
9. Update failing Jest tests
